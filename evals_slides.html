Evals: 为 AI 定义“好”的标准

Evals 是衡量 AI 系统质量和效果的方法。它为你的 AI 产品清晰地定义了「好」的标准。

AI Evals 像驾照考试

  • 感知: 能否准确解读信号？
  • 决策: 能否稳定做出正确选择？
  • 安全: 能否始终遵循指令？

传统单元测试像检查火车

  • 过程直接
  • 结果确定
  • 通过/失败界限分明

Evals 的三种方法

人工 Evals

核心: 人工反馈或专家标注 (如 👍/👎)。

优点: 黄金标准，最贴近用户体验。

缺点: 慢、贵、数据稀疏。

基于代码的 Evals

核心: 检查代码执行结果或 API 调用。

优点: 快速、便宜、确定性强。

缺点: 难以评估主观或开放式任务。

基于 LLM 的 Evals

核心: 使用另一个 LLM 作为“裁判”来打分。

优点: 可扩展性强，能评估复杂任务。

缺点: 需要初始校准，结果是概率性的。

AI Agent 评估流程分解

用户: “帮我订去旧金山的行程”
1. 工具选择
出错: 调用错误工具
评估: 工具选择 Eval
↓
2. API 搜索
出错: 搜索指令错误
评估: 函数调用 Eval
↓
3. 上下文利用
出错: 调用错误工具
评估: RAG Eval
↓
4. 回复构建
出错: 语气或内容不当
评估: 语气 Eval
↓
5. 最终正确性
出错: 整体方案错误
评估: 正确性 Eval
最终结果: 订了去圣地亚哥的票

对AI决策的每个环节设立“质检站”（即各种Eval），是避免“差之毫厘，谬以千里”的关键。 代表使用 LLM-as-a-judge 方法。

通用评估标准有哪些？

幻觉 (Hallucination)

AI Agent 是准确利用了上下文，还是在凭空捏造？

恶意/语气 (Toxicity/Tone)

输出是否包含有害或不当言论？

总体正确性 (Overall Correctness)

系统在其核心目标上的表现如何？（如：问答准确率）

其他常见领域

代码生成、摘要质量、检索相关性等。

第一阶段：收集数据

核心任务

  • 收集真实用户交互： 捕获真实案例，例如用户留下的“赞/踩”反馈。
  • 记录边缘案例： 识别特殊或意外行为，以及 AI 的异常响应。
  • 构建代表性数据集： 整理成结构化数据集，初期准备 10-100 个带有人工标签的样本作为基准。

数据表示例

用户输入 AI 输出 人工标签
"周末去旧金山" "预订圣地亚哥的票" 错误
"最近的咖啡店" "500米外有星巴克" 正确

第三阶段：迭代循环

流程详解（按顺时针方向）

持续改进循环

1. 收集生产数据

从线上真实服务的AI模型中，收集实际的交互数据，反映真实世界表现。

2. 添加人工标注

人工审查和分类数据，例如标注正确、错误、或有害的AI回答。

3. 创建失败模式数据集

将AI“犯错”的案例整理归类，形成专门的数据集，作为改进的靶心。

4. 编写你的评估方案

针对失败模式设计和编写具体的评估方案（Eval），以稳定检测特定错误。

5. 在数据集上测试评估方案

先用Eval测试已知的失败数据集，确保你的Eval本身是有效的、可靠的。

6. 迭代你的智能体和评估方案

利用Eval结果找出弱点，修复问题，并同步优化评估方案。

第四阶段：生产环境监控

持续改进

  • 持续评估： 设置自动化流程，让 Evals 在实时用户交互中持续运行。
  • 对比真实反馈： 寻找评估结果与真实世界表现之间的差异。
  • 构建可指导行动的仪表盘： 将 AI 性能指标与业务成果挂钩。

Evals 设计要避免哪些错误？

起步时设计得过于复杂

会产生“噪音”信号，应先专注于具体输出。

不测试边缘案例

使用小样本提示可以显著提升性能。

忘记用真实用户反馈来验证

最终目标是解决用户的问题。

快速上手评估：三步走

1. 选择一个关键特性

例如“幻觉检测”

→

2. 编写简单 Eval

→

3. 复盘并迭代

总结

Evals 不仅是为了发现 bug，更是为了确保你的 AI 系统能够持续创造价值，赢得用户的喜爱。

Evals 是生成式 AI 从原型走向成熟产品的关键一步。
