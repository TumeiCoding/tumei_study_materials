2.1 深入分析生产数据 (Data)

核心论点：对生产环境中真实、动态的数据进行持续、深入的观察，是保证产品质量和迭代方向的唯一可靠途径。

"summarize this""sumarize thiss plz!"

每天随机抽样检查生产日志，快速建立对模型行为的直觉，发现意想不到的问题。

4

在测试中考虑不确定性

对同一个测试输入多次运行，以提高捕捉间歇性异常的概率。

// Pseudocode for testing nondeterminism

const prompt = "Is the sky blue?";

for (i = 0; i < 5; i++) {

output = call_llm(prompt);

assert(is_valid(output)); // May fail on 3rd run

}

观点 💡

⚠️ “数据偏差”是新瓶颈

指出数据环境不一致是产品失败的核心根源，即使我们不训练模型。

📈 评估标准动态演化

对“好/坏”的判断会变。评估体系必须是一个与产品一同迭代的生命体。

V1: "不崩溃就行"

↓

V2: "回答必须是事实"

↓

V3: "回答要简洁有礼貌"

🔬 “现场现物”的哲学

强调生产数据是唯一的真相来源，任何模拟数据都无法取代对真实用户交互的直接观察。

💡→⚙️ 从“直觉”到“系统”

描绘了从主观“感觉不对”到客观自动化代码的路径，连接人类直觉和工程严谨性。

2.2 与模型共事 (Working with models)

核心论点：在使用不受你完全控制的第三方LLM API时，如何通过工程实践保持应用的稳定、高效和经济。

行动指南 🧰

1

强制输出结构化数据

使用 JSON/YAML 等机器可读格式，确保下游程序的稳定可靠。

❌ 纯文本 (不可靠)

"用户信息：名叫张三，年龄是二十岁。"

→

✅ JSON (可靠)

{"user": {"name": "张三", "age": 20}}
2

锁定并管理模型版本

明确指定模型版本号，防止因后台更新导致生产环境行为突变。

// 浮动别名 (危险)

model = "gpt-4-turbo"

// 锁定版本 (安全)

model = "gpt-4-turbo-2024-04-09"

最佳实践：影子管道 (Shadow Pipeline)

生产流量 → [当前模型: v1.0] → 用户

| (复制)

→ [新模型: v1.1] → 记录/对比 (不影响用户)

3

优先选择能完成任务的最小模型

在满足需求的前提下，追求成本、速度和效率的最优化。

模型 能力 速度 成本
大型模型 (GPT-4)⭐⭐⭐⭐⭐🐢💰💰💰💰💰
中型模型 (Sonnet)⭐⭐⭐⭐🚀💰💰💰
小型模型 (Haiku)⭐⭐⭐⚡️💰
4

严肃对待模型迁移

更换模型时必须重新测试和调整Prompt，并建立自动化评估流程。

Prompt V1

"Summarize: {text}"

→

Claude 2

✅ 效果很好

↓

Prompt V1

→

Claude 3

❌ 悄悄失败

观点 💡

⛓️ Prompt的脆弱性

Prompt与特定模型版本高度耦合，其价值非常脆弱，不可轻易移植。

👍 “够用就好”的实用主义

反对“能力越强越好”，强调在满足需求的前提下，追求成本和效率的最优化。

🌊 “流程工程”是未来

相比依赖单一巨大模型，通过巧妙组合多个小模型或设计工作流是更优的长期方案。

🛡️ 风险内部化管理

通过版本锁定、影子测试等内部工程实践，管理和规避对外部模型的依赖风险。

🎯

2.3 产品 (Product)

核心论点：回归经典、以用户为中心的产品原则，是打造成功AI产品的关键。

行动指南 🧰

1

尽早并频繁地让设计师参与

从项目开始就让设计师深度参与，帮助团队定义正确的问题，而不是盲目套用AI。

设计师会问的关键问题：

“解决这个问题，聊天机器人是最佳形式吗？还是自动补全？或者其他？”

2

设计一个“人机回圈”的用户体验

将产品交互设计成“AI建议，用户确认”的模式，既能提升用户体验，又能高效收集反馈数据。

🤖

AI建议

→
🧑‍💻

用户确认/修改

→
📊

高质量反馈

3

冷酷地确定需求的优先级

用“需求层次”模型定义“最小可爱产品”，接受第一版的不完美，快速发布并迭代。

可爱功能
核心特性
可靠性
安全性
4

根据用例来校准你的风险容忍度

评估应用场景和受众，决定对准确性和安全性的要求，先从内部低风险应用做起。

高风险场景

医疗/金融建议、面向客户

要求: 99.9% 准确

低风险场景

内部摘要、内容分类

要求: 80% 准确即可

观点 💡

🛠️ 技术是手段，非目的

回归产品设计的黄金法则：“这个产品帮用户解决了什么核心任务？”

🔄 UX是数据引擎

好的UX设计能让产品自身变成一台高效、自动、隐形的数据标注机器。

🔺 用“需求层次”做规划

将复杂要求变成有结构、有优先级的战略地图，指导产品如何分阶段推向市场。

🚶→🏃 “先内后外”的务实路径

先用LLM赋能内部效率，在低风险“练兵场”学习，再进入高风险“主战场”。

👥

2.4 团队与角色 (Team & Roles)

核心论点：正确的思维方式和团队协作，远比追逐时髦的工具和头衔更重要。

行动指南 🧰

1

流程优于工具

先理解问题和流程，再选择或自研工具，避免不必要的技术债。

🔧

2

“评估”是“实验”的加速器

评估体系的核心价值是赋能快速、低成本的实验，从而加速产品进化。

强大评估

→

快速实验

→

产品进化

3

赋能全员，而非专家垄断

通过内部教育和实践，让整个团队都理解并能动手使用LLM，能极大地拓宽创新来源。

👩‍💻

🎨

4

按阶段组建跨职能团队

根据产品成熟度，按顺序引入不同角色，而不是指望“AI工程师”是万能的。

初期 (验证): AI工程师 + 产品经理
中期 (基建): + 平台/数据工程师
后期 (优化): + 机器学习工程师

观点 💡

🧠 流程是本源

真正的竞争力来自于对问题解决流程的深刻理解，工具只是流程的固化产物。

🚀 评估是创新引擎

评估的战略价值是驱动创新：评估能力 → 实验速度 → 产品进化速度。

🌐 AI能力民主化

将LLM能力扩散到整个组织，能极大拓宽创新来源，放大集体智慧。

🔄 角色演进的历史周期

“AI工程师”也将从被神化回归理性，伟大的产品永远依赖于协作的团队。
